Start here

Your local sprinkler controller

YardRelay runs on your computer and connects to the Orbit B-hyve account you already use. It gives you one place to view controllers and zones, start or stop watering, review history, manage supported program settings, and run configured groups of zones in sequence.

Before watering: confirm the controller, zone, and run time. A manual start can turn on real irrigation hardware immediately.

Overview

Getting started

  1. Complete platform setup. Use the Windows setup screen or the current macOS source-checkout steps below.
  2. Start the controller server. Desktop apps start it automatically after setup. A source checkout uses npm start.
  3. Wait for a healthy connection. The top of the dashboard shows controller and Orbit connection status.
  4. Review before changing anything. Confirm controller names, zones, programs, and current watering before using a write control.
The supported deployment is one trusted user on one computer through the loopback interface. It is not designed for LAN or public-internet access.

Current capabilities

Local dashboard
View controllers, zones, active watering, upcoming schedules, watering history, events, and sanitized logs in one local interface.
Manual watering
Start and stop individual zones with configurable run-time caps and safeguards against conflicting known activity.
Program controls
Enable, disable, start, and edit supported fields on existing Orbit programs while keeping smart programs read-only.
  • Program changes are snapshotted locally before the Orbit API is updated.
Named yard runs
Run locally configured groups such as a front yard, garden, or planter sequence one zone at a time.
  • Adjust minutes per zone before starting.
  • Queue additional runs without watering shared zones twice.
  • Stop the active zone and clear the remaining queue from one control.
  • Recover the active queue display after a local server restart.
Local browser write access
Keep same-computer loopback controls prompt-free or require the app token for every browser.
  • The older trusted-network mode remains for private-development compatibility and is not a supported public deployment.
Optional desktop apps
Run the same local controller in a macOS SwiftUI wrapper or a Windows WPF and WebView2 wrapper.
  • Both wrappers verify the local controller before loading its embedded dashboard.
  • Windows includes a per-user setup flow with encrypted credential storage and a read-only Orbit login test.

Setup

Windows setup

The Windows app opens Desktop setup automatically when required settings or credentials are missing.

1. Check prerequisites

  • Windows 11 on an x64 computer.
  • Node.js 24 or newer.
  • Microsoft Edge WebView2 Runtime.

Download Node.js Download WebView2

2. Enter account and server settings

  • Enter the email address and password for your Orbit B-hyve account.
  • Keep port 3030 unless another local app already uses it.
  • Leave Node path empty when the app detects a suitable Node.js installation.

3. Test and save

  1. Select Test Orbit Login. The test reads account metadata and does not start watering or change programs.
  2. Optionally import an existing .env file or yard-run configuration when migrating from a source checkout.
  3. Select Save and start.
A failed login test does not always mean the credentials are wrong. Orbit or the internet connection may be temporarily unavailable. You can save and retry later.

Setup

macOS setup

The current macOS wrapper uses the source-checkout configuration flow. It does not yet have the Windows-style first-run setup wizard.

  1. Install Node.js 24 or newer. Confirm it is available by running node --version in Terminal.
  2. Install project dependencies. From the project folder, run npm ci.
  3. Create local configuration. Copy .env.example to .env, then add your Orbit email, password, and a long random app token.
  4. Build or run the wrapper. Run swift build --package-path mac/BHyveControllerApp, or use the app-bundle build script described in the project README.
Keep .env private. Never attach it to an issue, include it in a screenshot, or commit it to Git.

Daily use

Dashboard

Connection status
Shows whether the local server is reachable, Orbit authentication succeeded, and the live event stream is connected.
Active watering
Appears when a zone is running and shows the current zone, remaining time, and a Stop control.
Upcoming schedule
Summarizes scheduled watering expected during the next 48 hours. Expand it to review more entries.
Controller cards
Show controller mode, rain delay, zones, and manual watering controls.
Refresh
Requests current data from Orbit. Refreshing does not start or stop watering.

Daily use

Manual watering

Start a zone

  1. On Dashboard, find the correct controller and zone.
  2. Enter the number of minutes. The value must be within the manual run cap set under Settings.
  3. Select Start and confirm the zone begins reporting as active.

Stop a zone

Use Stop in the Active watering panel or beside the active zone. Wait for the dashboard to report that watering ended. If the status does not update, select Refresh before trying another start.

The controller blocks conflicting manual starts when it knows another zone is active. Always verify the physical sprinkler state if the network connection is interrupted.

Daily use

Watering history

History reads recent watering activity from Orbit and organizes it by day, controller, zone, and duration. Choose a 24-, 72-, or 184-hour window, or enter another supported number of hours.

  • Select a day in the calendar to view its events.
  • Use Show details to expand the event panel.
  • Select Refresh when recent watering is not yet visible.
Orbit history can arrive later than live controller status. A missing recent event does not by itself mean the watering action failed.

Programs

Programs

The Programs tab works with programs that already exist in Orbit.

  • Enable or Disable changes whether a supported program runs on schedule.
  • Start now immediately starts a supported non-smart program.
  • Edit schedule changes supported fields such as budget, frequency, and start times.
  • Smart programs are shown read-only.
Before a program update is sent to Orbit, the local server saves a snapshot under its data folder. The app does not currently create or delete Orbit programs.

Programs

Yard runs

A yard run is a locally configured sequence of zones, such as a front yard, garden, or planter group. Yard runs appear only when a local recipe file defines them.

  1. Open Programs and find the yard-run controls.
  2. Review the run name, zones, and minutes per zone.
  3. Select the watering button. The server runs one zone at a time and shows Now and Next.
  4. Select Stop yard run to stop the current zone and clear the remaining queue.
  • Adding another run while one is active appends its unique zones to the queue.
  • Shared zones are deduplicated so they are not watered twice in the same queue.
  • The local server saves active queue state so the display can recover after a restart.
Yard-run recipes describe a specific property and may contain private controller identifiers. Keep the local recipe file out of Git, screenshots, support posts, and release packages.

Configuration

Controller settings

The Settings tab inside the dashboard controls browser-level behavior. The desktop Settings button on Windows opens account, port, Node.js, import, diagnostics, and reset options.

Manual run cap
Sets the largest manual zone or yard-run duration the dashboard will accept, from 0 to 120 minutes. Set it to 0 to disable manual starts in the browser.
Access and writes
Explains that the controller is local and that program changes are snapshotted before Orbit is updated.

Recovery

Troubleshooting

The controller server will not start
  • Confirm Node.js 24 or newer is installed and detected.
  • Check whether another app is using the configured port. Try another port between 1024 and 65535.
  • On Windows, open Settings and use Copy diagnostics. Diagnostics omit credentials and tokens.
  • Review the wrapper message and sanitized local logs for the first reported error.
Windows reports that WebView2 is unavailable

Install or repair Microsoft Edge WebView2 Runtime, close the controller app, and open it again. Open the official WebView2 download page.

Orbit login fails or the dashboard stays disconnected
  • Confirm the computer has internet access and Orbit services are reachable.
  • Re-enter the same email and password used by the official B-hyve account.
  • Use Test Orbit Login on Windows. The test is read-only.
  • If credentials recently changed, restart the controller after saving them.
Write controls are locked in a normal browser

Select Unlock writes and enter the local APP_TOKEN. Native wrappers unlock their verified embedded dashboard automatically. Never share the token or put it in a URL query string.

Yard-run controls are missing

Yard runs are optional. Import or create a valid local yard-run recipe, then restart the server. Basic controllers, zones, and Orbit programs work without a yard-run file.

History or live status looks stale

Select Refresh and wait for the connection badges to recover. History may lag behind live status. If the network was interrupted, verify the physical sprinkler state before issuing another command.

I need to reset Windows setup

Open Settings, select Reset setup, and confirm. This stops the managed server and removes saved desktop settings and encrypted credentials for the current Windows user. Yard-run configuration, snapshots, logs, and active-run data are kept.

Important

Privacy and safety

  • Keep the controller bound to loopback unless you have deliberately configured and secured private-network access.
  • Never expose the Node server directly to the public internet.
  • Keep Orbit credentials and APP_TOKEN out of Git, screenshots, issue reports, and chat messages.
  • Do not publish device identifiers, property-specific zone names, schedules, logs, or yard-run recipes.
  • Use Copy diagnostics when available; it is designed to omit credentials, tokens, email addresses, device identifiers, and raw request data.
  • Verify the real sprinkler state after connection failures, restarts, or unexpected controller behavior.

Advanced

Source checkout and local files

For source-checkout use, install dependencies with npm ci, configure .env, and run npm start.

Common environment settings

ORBIT_EMAIL / ORBIT_PASSWORD
Orbit account credentials. Keep them backend-only.
HOST
Use 127.0.0.1 for the default local-only deployment.
PORT
Local controller port, normally 3030.
APP_TOKEN
A long random local token required for protected browser writes and desktop service verification.
BHYVE_DATA_DIR
Optional desktop path for snapshots and active yard-run state.
YARD_RUN_CONFIG
Optional path to the local property-specific yard-run recipe.

Where desktop data lives

  • Windows private-development builds: settings, config, data, and logs currently remain under the legacy %LOCALAPPDATA%\BHyveController path. Public builds will migrate to %LOCALAPPDATA%\YardRelay only after migration tests are complete. Secrets are encrypted for the current Windows user.
  • macOS today: the wrapper reads the source checkout and its private .env. Native Keychain and Application Support parity is planned separately.
